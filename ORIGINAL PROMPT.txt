You are Claude Code. Build a production-ready digital signage software system for Raspberry Pi and ODROID-C4 devices.

The product must allow users to manage signage screens remotely without setting up VPN, port forwarding, static IPs, or on-site router changes. Devices must connect outbound to the cloud service and stay controllable through that outbound connection.

The system must support scheduling of image and video content, reliable playback, portrait and landscape screens, offline playback, and remote device monitoring.

Build this as a complete monorepo with clear setup instructions, runnable development environment, database migrations, API docs, tests, and deployment scripts.

## 1. Product summary

Create a digital signage platform with three main parts:

1. Cloud backend
   A server that handles users, organizations, devices, media, schedules, playlists, device commands, device status, and content synchronization.

2. Web admin dashboard
   A browser-based dashboard where users can upload media, create playlists, schedule content, register devices, monitor devices, and control screens remotely.

3. Linux device player/agent
   Software that runs on Raspberry Pi and ODROID-C4. It must boot into fullscreen signage playback, connect outbound to the backend, download assigned content, cache it locally, play scheduled media, report status, and recover automatically after network loss, reboot, or crashes.

## 2. Hardware and operating system targets

Target devices:

* Raspberry Pi 4 and Raspberry Pi 5
* ODROID-C4
* ARM64 Linux preferred
* Raspberry Pi OS 64-bit for Raspberry Pi
* Ubuntu/Debian-based Linux for ODROID-C4

The player must be designed for low-power ARM devices and must avoid heavy desktop dependencies where possible.

## 3. Core requirements

### Remote connectivity

The device must never require inbound connections.

Implement an outbound-only connection model:

* Device initiates connection to cloud backend over HTTPS and secure WebSocket.
* Device maintains a persistent WebSocket connection for commands and status updates.
* If WebSocket fails, device falls back to periodic HTTPS polling.
* Backend can send commands through the existing outbound connection.
* No VPN.
* No port forwarding.
* No static IP requirement.
* Devices behind NAT/firewalls must still work as long as outbound HTTPS/WSS traffic is allowed.

Required remote commands:

* Restart player
* Reboot device
* Refresh content
* Clear cache
* Take screenshot if technically available
* Identify device
* Change orientation
* Change assigned playlist/schedule
* Update device settings
* Show emergency content
* Stop emergency content
* Send logs
* Run health check
* Trigger software update placeholder/hook

### Device pairing

Implement secure device registration:

* Admin creates a device in the dashboard.
* System generates a short pairing code and/or QR code.
* Device setup flow asks for server URL and pairing code.
* Device calls backend to pair.
* Backend returns a unique device ID and long-lived device token.
* Store token securely on the device.
* Device authenticates future requests using its token.
* Pairing codes must expire.
* Pairing codes must be single-use.
* Admin can revoke a device token.

### Media upload and processing

Support uploading:

* Images: JPG, JPEG, PNG, WebP
* Videos: MP4, MOV, MKV, AVI, WebM, M4V and other common FFmpeg-supported formats

Important: playback must be reliable on Raspberry Pi and ODROID-C4. Do not assume every uploaded video format can be played natively. Implement a media processing pipeline that accepts many formats and converts them into a signage-safe playback format.

Use FFmpeg/ffprobe for:

* Inspecting media metadata
* Detecting duration
* Detecting resolution
* Detecting orientation
* Generating thumbnails
* Transcoding unsupported or non-optimal videos
* Normalizing videos to H.264 video + AAC audio inside MP4 container
* Creating a lower-bitrate fallback version if needed

Store original media and processed media separately.

Media fields:

* id
* organization_id
* original_filename
* media_type: image/video
* original_mime_type
* processed_mime_type
* original_url/storage_key
* processed_url/storage_key
* thumbnail_url/storage_key
* duration_seconds
* width
* height
* orientation: portrait/landscape/square
* processing_status: pending/processing/ready/failed
* processing_error
* created_at
* updated_at

### Portrait and landscape behavior

The system must fully support both portrait and landscape displays.

Important behavior:

* Portrait images are uploaded as real portrait files.
* Landscape images are uploaded as real landscape files.
* Do not require users to rotate the media manually after upload.
* Detect media width and height.
* Automatically classify media as portrait, landscape, or square.
* Devices have a configured screen orientation:

  * landscape
  * portrait
  * inverted_landscape
  * inverted_portrait
* The device player must render content correctly for the configured screen orientation.
* The admin dashboard must show whether content is portrait or landscape.
* Playlists and schedules should indicate if media orientation does not match the target device orientation.
* Provide an option to either:

  * block mismatched content from being assigned, or
  * allow it with a fit mode.

Fit modes:

* contain: show full image/video, possible black bars
* cover: fill screen, crop overflow
* stretch: fill screen without preserving aspect ratio
* original: center at original aspect ratio

Default fit mode: contain.

Device orientation must be configurable remotely from the dashboard.

### Scheduling

Build a flexible scheduling system.

Required schedule types:

1. Always active playlist
2. Date range schedule
3. Weekly recurring schedule
4. Time-of-day schedule
5. Priority/emergency override

Examples:

* Play playlist A every weekday from 09:00 to 17:00.
* Play playlist B every Saturday and Sunday.
* Play campaign C from 2026-07-01 through 2026-07-31.
* Play emergency message immediately on selected devices.
* Fall back to default playlist if no schedule matches.

Scheduling rules:

* Each device can have one or more assigned schedules.
* Each schedule points to a playlist.
* A playlist contains ordered media items.
* Each playlist item has duration.
* Images require a configurable display duration.
* Videos should play to their natural duration by default.
* Allow overriding video duration.
* Allow per-item fit mode.
* Allow per-playlist loop mode.
* Allow schedule priority.
* If multiple schedules match, highest priority wins.
* Emergency override always wins.
* Device must be able to calculate schedule locally after syncing so playback continues offline.

Timezone handling:

* Each device has a timezone.
* Schedules are interpreted in the device timezone unless specified otherwise.
* Store times in a robust format.
* Avoid bugs around DST.
* Display timezone clearly in the dashboard.

### Offline playback

The device must keep playing when internet is unavailable.

Required behavior:

* Device downloads and caches all currently required media.
* Device stores schedules, playlists, and media metadata locally.
* Device continues playing the last valid schedule offline.
* Device keeps a local playback queue.
* Device retries failed downloads.
* Device reports sync status when back online.
* Device must never show a blank screen just because internet is down.
* If no content is cached, show a local fallback screen with device name, pairing/status info, and error message.

Local storage:

* Use a local SQLite database on the device for synced metadata.
* Store media files in a local cache directory.
* Maintain cache manifest with checksum/size.
* Delete unused old media only after confirming new content is downloaded.
* Never delete currently playing media.

### Playback engine

Implement a robust player for ARM Linux.

Preferred approach:

* Device agent written in TypeScript/Node.js.
* Fullscreen kiosk UI rendered in Chromium or a lightweight browser shell.
* Playback UI uses HTML/CSS/JavaScript with image and video elements.
* Device agent controls the browser/player and local cache.
* Use systemd services to keep both agent and player running.
* Use watchdog/restart behavior.

Alternative if needed:

* Use mpv/VLC as a fallback player for problematic videos, but the main reliable path should be processed MP4 playback in the browser.

Playback requirements:

* Smooth transitions between items.
* No visible desktop.
* Fullscreen on boot.
* Hide cursor.
* Disable screen blanking.
* Disable power saving.
* Recover after crash.
* Resume after reboot.
* Avoid browser cache dependency; use local file paths or local HTTP server.
* Preload next media item.
* Log playback start/end/errors.
* Report current playing media to backend.
* Support portrait/landscape screen rendering.
* Support image duration.
* Support video natural duration.
* Support muted autoplay by default.
* Add future option for audio enable/disable.

### Sync protocol

Design a versioned sync protocol.

Device should call sync with:

* device_id
* current local manifest version
* app version
* current media cache checksums
* free disk space
* capabilities
* current orientation
* current timezone

Backend responds with:

* sync version
* device settings
* active schedules
* playlists
* media manifest
* download URLs
* checksums
* emergency override state
* commands if polling fallback is active

Use checksums to avoid unnecessary downloads.

Device applies sync transactionally:

1. Download new media to temp files.
2. Verify checksum.
3. Store metadata in SQLite.
4. Mark manifest ready.
5. Switch playback to new manifest.
6. Cleanup unused old files later.

### Sync protocol

Design a versioned sync protocol.

Device should call sync with:

* device_id
* current local manifest version
* app version
* current media cache checksums
* free disk space
* capabilities
* current orientation
* current timezone

Backend responds with:

* sync version
* device settings
* active schedules
* playlists
* media manifest
* download URLs
* checksums
* emergency override state
* commands if polling fallback is active

Use checksums to avoid unnecessary downloads.

Device applies sync transactionally:

1. Download new media to temp files.
2. Verify checksum.
3. Store metadata in SQLite.
4. Mark manifest ready.
5. Switch playback to new manifest.
6. Cleanup unused old files later.
